# 半自动桌面 MVP 规格说明

## 1. 产品目标

QNU Thesis Copilot 的第一阶段不是全自动生成论文，而是把已经跑通的人机协作流程封装成一个稳定的桌面工具，帮助用户完成以下事项：

- 收集和整理参考文献 PDF
- 固定使用 `NotebookLM` 完成依赖文献事实的大纲和正文生成
- 使用任意通用 LLM 完成文献推荐、切块规划、上下文脱水和格式修复
- 解析并校验 AI 返回的 JSON
- 将结构化内容注入青海师范大学固定模板，导出格式合规的 `.docx`

## 2. 目标用户

- 第一批用户：项目作者本人、朋友、同专业同学
- 第二批用户：青海师范大学校内学生

本阶段默认用户不懂代码，也不熟悉 Prompt Engineering，因此每一步都必须提供明确教学。

## 3. 本阶段边界

### 3.1 In Scope

- Windows 优先的桌面端应用
- 本地工作区和本地数据持久化
- 单项目流程引导
- 固定青海师范大学模板导出
- 文献推荐的半自动流程
- PDF 拖拽、重命名、排序和跳过补位
- 大纲 JSON 解析与树编辑
- 切块规划 JSON 解析
- 正文块的逐块生成与保存
- 引用占位符还原与参考文献自动生成

### 3.2 Out of Scope

- 在线账户系统
- 多用户协作
- 云端 API 调用
- 自定义学校模板上传
- 自动下载论文 PDF
- 自动打开或控制 NotebookLM 页面
- 面向所有高校的开放平台化能力

## 4. 核心产品原则

### 4.1 模型能力边界

- 必须使用 `NotebookLM` 的步骤：
  - 基于已上传 PDF 生成大纲
  - 基于已上传 PDF 生成正文块
- 可使用任意通用 LLM 的步骤：
  - 文献推荐与 BibTeX 汇总
  - 切块规划
  - 上下文脱水压缩
  - JSON 修补和轻量格式修复

### 4.2 软件是调度器，不是自由聊天入口

用户不能直接在软件里和 AI 对话。软件只负责：

- 生成 Prompt
- 展示教学文案
- 接收并校验 AI 回传 JSON
- 驱动流程进入下一步

### 4.3 强约束优先于灵活性

本阶段优先保证成功率，不优先保证自由度：

- 固定模板
- 固定阶段
- 固定数据协议
- 固定解锁条件

## 5. 端到端流程

## 5.1 阶段 0：新建项目

用户输入：

- 论文题目
- 核心思想摘要
- 学科方向或关键词
- 是否需要系统推荐参考文献
- 可选最低总字数

系统动作：

- 创建工作区目录
- 初始化 `state.json`
- 生成项目唯一 ID

解锁条件：

- 题目非空
- 核心思想达到最小长度要求

## 5.2 阶段 1：参考文献来源分流

### 路线 A：需要系统推荐参考文献

系统生成文献检索 Prompt，明确要求用户把该 Prompt 发送给支持深度研究的通用 LLM，并返回严格 JSON，内容包括：

- 中文论文至少 15 篇
- 英文论文至少 15 篇
- 每篇论文的题目
- 下载链接
- BibTeX
- 语言
- 影响力或推荐理由

用户把 AI 返回 JSON 粘贴回软件。

软件动作：

- 清洗 Markdown 包裹
- 校验 JSON Schema
- 展示论文列表与下载链接
- BibTeX 只存储，不在该界面默认展开

### 路线 B：已有参考文献

系统跳过推荐环节，要求用户：

- 准备已有 PDF
- 准备 BibTeX
- 确保 PDF 文件名与论文标题能对应

解锁条件：

- 路线 A：推荐 JSON 校验通过
- 路线 B：用户确认已准备现有文献

## 5.3 阶段 2：PDF 资产处理

### 路线 A：逐篇下载模式

界面一次只展示一篇待处理论文：

- 标题
- 语言
- 下载链接
- 当前序号
- 拖拽上传区
- “不可下载，跳过此篇”按钮

规则：

- 用户拖入 PDF 后，软件复制文件到工作区
- 软件按清洗后的标题重命名
- 文件名前缀使用连续编号，如 `01_论文标题.pdf`
- 若某篇跳过，则编号不占位，下一篇递补

### 路线 B：批量导入已有 PDF

用户一次性拖入多个 PDF。

软件动作：

- 尝试按文件名匹配论文标题
- 对已匹配成功的 PDF 重新编号并复制到 `processed_pdfs`
- 对未匹配项给出人工确认

解锁条件：

- 至少满足系统设定的最低文献数量
- 所有待处理条目都被标记为“已导入”或“已跳过”

## 5.4 阶段 3：大纲生成

系统根据以下信息生成大纲 Prompt：

- 论文题目
- 核心思想
- 已整理参考文献清单
- 固定的大纲生成模板

教学文案必须明确要求：

- 本步骤必须去 `NotebookLM`
- 用户需要先把 `processed_pdfs` 中的 PDF 导入 `NotebookLM`
- 用户需要在 `NotebookLM` 中设置软件给出的系统提示词
- 用户必须返回严格 JSON，不接受散文式回复

用户粘贴返回结果后，软件动作：

- 去除 Markdown 代码块
- 容错修复尾部括号
- 按大纲 JSON Schema 校验
- 存入 `state.json`

解锁条件：

- 大纲 JSON 校验通过

## 5.5 阶段 4：大纲树编辑

软件把大纲显示为可编辑树结构，支持：

- 新增同级/下级节点
- 删除节点
- 拖拽调整顺序
- 修改标题
- 标记某节点是否必须单独成块

解锁条件：

- 至少形成完整三级结构或用户手动确认采用当前版本

## 5.6 阶段 5：切块规划

系统根据以下信息生成切块规划 Prompt：

- 最终大纲树
- 目标最低总字数
- 固定的切块规划模板

本步骤允许任意通用 LLM。

返回 JSON 需至少包含：

- 建议总块数
- 每块覆盖的大纲节点
- 每块建议最小字数
- 每块写作目标
- 每块需要重点引用的文献编号范围或文献主题

系统动作：

- 校验 JSON
- 检查所有大纲节点是否被覆盖
- 检查块顺序是否连续

解锁条件：

- 分块方案覆盖全部启用节点

## 5.7 阶段 6：正文逐块生成

每一块都走固定循环：

1. 软件生成“当前块正文生成 Prompt”
2. 用户复制到 `NotebookLM`
3. 用户把返回的块级 JSON 粘贴回软件
4. 软件校验并保存该块
5. 软件基于全部已生成块，生成“上下文脱水 Prompt”
6. 用户复制到任意通用 LLM
7. 用户把脱水结果粘贴回软件
8. 软件保存压缩上下文，并组装下一块 Prompt

设计约束：

- 通用 LLM 只负责返回结构化 `compressed_context`
- 下一块发给 `NotebookLM` 的最终 Prompt 必须由软件本地模板拼装
- 用户不得手工编辑由软件生成的中间结构数据

解锁条件：

- 当前块正文 JSON 校验通过
- 如不是最后一块，则当前轮压缩 JSON 校验通过

## 5.8 阶段 7：最终导出

系统动作：

- 汇总所有块内容
- 按最终有效文献顺序重映射 `【文献XX】`
- 将参考文献转换为 GB/T 7714
- 将正文注入学校模板
- 生成 `.docx`

输出：

- 导出文件路径
- 生成日志
- 可选中间 JSON 快照

## 6. 页面清单

建议 MVP 至少包含以下页面或向导步骤：

1. 欢迎页 / 工作区选择
2. 新建项目页
3. 参考文献来源选择页
4. 推荐文献结果导入页
5. 单篇 PDF 下载处理页
6. 批量 PDF 导入页
7. BibTeX 导入页
8. 大纲生成教学页
9. 大纲粘贴与解析页
10. 大纲树编辑页
11. 切块规划教学页
12. 切块方案粘贴与解析页
13. 块生成工作台
14. 导出页

## 7. 关键状态与解锁逻辑

### 7.1 阶段前后依赖

- 没有完成 PDF 资产处理，不能进入大纲生成
- 没有完成大纲解析，不能进入大纲编辑
- 没有完成大纲编辑确认，不能进入切块规划
- 没有完成切块规划，不能进入正文生成
- 没有完成全部块生成，不能进入最终导出

### 7.2 可恢复性

用户关闭软件后再次打开，必须能：

- 自动读取 `state.json`
- 恢复到上次所在阶段
- 显示每一步是否完成、失败或待修复

## 8. 异常与容错

### 8.1 AI 返回格式异常

需要容忍：

- Markdown 代码块包裹
- 前后多余解释文字
- 尾部截断导致的 `]` 或 `}` 缺失
- 数字编号字符串化

不容忍：

- 关键字段缺失
- 节点类型不合法
- 切块方案存在未覆盖节点

### 8.2 文件异常

需要处理：

- PDF 名称中包含非法文件字符
- 同名论文
- 用户拖入非 PDF 文件
- PDF 已存在但内容不一致
- 标题过长导致路径超长

### 8.3 用户操作异常

需要处理：

- 未按要求先去 `NotebookLM`
- 粘贴了错误阶段的 JSON
- 跳过过多文献导致最低数量不足

## 9. 第一开发优先级

推荐实现顺序：

1. 工作区与 `state.json`
2. PDF 导入与重命名
3. BibTeX 存储与文献列表
4. 大纲 JSON 导入与树编辑
5. Word 导出引擎接入
6. 切块规划
7. 正文逐块循环
8. JSON 容错和恢复体验打磨
